1. MQTT Modes

Meshtastic firmware uses MQTT in two dimensions: transport and payload format. These dimensions are independent.

1.1 Transport Modes

ModeSettingDescription
Direct DIRECT proxy_to_client_enabled: false The radio connects to the MQTT broker using its own WiFi or Ethernet interface. The radio runs the MQTT client in its firmware and publishes/subscribes directly.
Client Proxy PROXY proxy_to_client_enabled: true A connected device (phone, tablet, web client) connects to the broker and forwards MQTT traffic for the radio. The radio's own MQTT client stays dormant.

1.2 Payload Formats

FormatSettingDescription
Protobuf (binary) PROTO json_enabled: false Encoded ServiceEnvelope protobuf messages. Raw binary payload on MQTT. Requires protobuf decoding to read. Default format.
JSON JSON json_enabled: true Human-readable JSON payload. Easier to integrate with external systems. Not available on nRF52 platform unless compiled with NRF52_USE_JSON.

2. Configuration Settings

Configure MQTT with the Meshtastic CLI. Chain multiple settings in one command to prevent a reboot between each setting.

# All MQTT settings
meshtastic --set mqtt.enabled true \
  --set mqtt.address mqtt.gwyn.tel \
  --set mqtt.username meshdev \
  --set mqtt.password '<pass>' \
  --set mqtt.root 'msh/US/bayarea' \
  --set mqtt.tls_enabled true \
  --set mqtt.encryption_enabled true \
  --set mqtt.json_enabled true \
  --set mqtt.proxy_to_client_enabled false \
  --set mqtt.map_reporting_enabled false
SettingTypeDefaultDescription
mqtt.enabledboolfalseTurns the MQTT module on or off.
mqtt.addressstringmqtt.meshtastic.orgMQTT broker hostname or IP address.
mqtt.usernamestringmeshdevMQTT broker username.
mqtt.passwordstringlarge4catsMQTT broker password.
mqtt.rootstringmshRoot topic prefix for all MQTT messages. Case-sensitive.
mqtt.tls_enabledboolfalseUses TLS to encrypt the connection. Port changes to 8883.
mqtt.encryption_enabledboolfalseEncrypts MQTT payload at the transport layer. When false, decoded MeshPackets are sent.
mqtt.json_enabledboolfalseEnables JSON uplink and downlink on MQTT. Required for JSON-based integrations.
mqtt.proxy_to_client_enabledboolfalseRoutes MQTT traffic through a connected device instead of connecting directly.
mqtt.map_reporting_enabledboolfalseSends periodic map reports to the MQTT broker. Available from firmware 2.3.2.

3. Channel Uplink and Downlink

Each channel has independent MQTT forwarding controls. At least one channel must have uplink or downlink enabled for the MQTT module to connect.

SettingDirectionDescription
uplink_enabled: trueRadio → BrokerThe radio forwards messages from this channel to the MQTT broker.
downlink_enabled: trueBroker → RadioThe radio receives messages from the broker and injects them onto this channel.
# Set per-channel MQTT settings
meshtastic --host <ip> --ch-index 0 --ch-set uplink_enabled true
meshtastic --host <ip> --ch-index 0 --ch-set downlink_enabled false
Warning: The CLI uses snake_case for channel settings (downlink_enabled), not camelCase (downlinkEnabled). The wrong case causes a silent failure.
Danger: Do not enable downlink on the primary channel (index 0) when connected to a community broker. Downlink on the primary channel injects internet-sourced traffic onto the local radio mesh. This floods the mesh with distant traffic.

4. MQTT Topic Structure

The firmware constructs MQTT topics from the mqtt.root setting. The binary segment is /2/c/ (for “crypt”).

{root}/2/c/{channelName}/{nodeId}    — Protobuf uplink
{root}/2/json/{channelName}/{nodeId} — JSON uplink
{root}/2/c/{channelName}/+           — Protobuf downlink subscription
{root}/2/json/{channelName}/+        — JSON downlink subscription
{root}/2/map/                        — Map reports
{root}/2/c/PKI/{nodeId}              — PKI-encrypted uplink
{root}/2/c/PKI/+                     — PKI downlink subscription

Channel Name in Topics

The channel name segment ({channelName}) comes from channels.getGlobalId(chIndex):

  • Unnamed primary channel (index 0): Uses the modem preset name (for example, LongFast or MediumFast).
  • Named channels: Uses the channel name string (for example, test, mqtt).
  • PKI-encrypted packets: Uses PKI as the channel name.

Example with root=msh/US/bayarea

Subscribe (protobuf):  msh/US/bayarea/2/c/mqtt/+
Subscribe (JSON):      msh/US/bayarea/2/json/mqtt/+
Subscribe (all ch):    msh/US/bayarea/2/c/#
Uplink (protobuf):     msh/US/bayarea/2/c/mqtt/!c1fc9198
Uplink (JSON):         msh/US/bayarea/2/json/mqtt/!c1fc9198
Publish JSON downlink: msh/US/bayarea/2/json/mqtt/<gateway_id>

5. JSON Downlink — Sending to the Mesh

JSON downlink sends text or position packets from the internet onto the mesh via MQTT.

Requirements

  1. mqtt.json_enabled must be true.
  2. A channel must be named "mqtt" (case-insensitive). The firmware checks this with strncasecmp.
  3. That channel must have downlink_enabled set to true.
  4. Publish JSON to the topic {root}/2/json/mqtt/<gateway_id>.

JSON Envelope Validation

The firmware validates incoming JSON with these rules (from isValidJsonEnvelope):

  • from — Required. Must be a number equal to the node's own decimal node number. This is a security check.
  • type — Required. Must be "sendtext" or "sendposition".
  • payload — Required. String for sendtext, object for sendposition.
  • sender — Optional. If present, must differ from the node's own ID (loop prevention).
  • hopLimit — Optional. Must be a number if present.

sendtext Schema

{
  "from": 3254555032,
  "type": "sendtext",
  "payload": "Hello mesh!",
  "channel": 2,
  "to": 4294967295,
  "hopLimit": 3,
  "sender": "!gw01"
}
FieldRequiredTypeDescription
fromYesnumber (uint32)Must equal the node's own decimal number. Security check.
typeYesstringMust be "sendtext" or "sendposition".
payloadYesstring/objectMessage text or position data.
channelNonumberChannel index (0-7). Defaults to 0 (primary).
toNonumberDestination node. 4294967295 = broadcast.
hopLimitNonumberRemaining hop count for radio forwarding.
senderNostringGateway ID for loop prevention.

sendposition Schema

{
  "from": 3254555032,
  "type": "sendposition",
  "payload": {
    "latitude_i": 377743000,
    "longitude_i": -1224194000,
    "altitude": 10,
    "time": 1689235200
  },
  "channel": 2,
  "to": 4294967295
}

6. JSON Uplink — Node to Broker

When json_enabled is true, the firmware publishes JSON-formatted messages for each packet it forwards to the broker.

{
  "id": 1234567,
  "timestamp": 1689235200,
  "to": 4294967295,
  "from": 3254555032,
  "channel": 2,
  "type": "text",
  "sender": "!c1fc9198",
  "rssi": -45,
  "snr": 9.5,
  "hops_away": 1,
  "hop_start": 3,
  "payload": {
    "text": "Hello from the mesh!"
  }
}
Note: If the text payload is itself valid JSON, the firmware places the raw JSON directly in "payload" instead of wrapping it in {"text": "..."}.

7. Protobuf (Binary) Format

The binary MQTT path uses the ServiceEnvelope protobuf message, published as raw bytes (not base64).

message ServiceEnvelope {
  MeshPacket packet     = 1;
  string channel_id     = 2;   // Channel name (e.g. "mqtt", "LongFast")
  string gateway_id     = 3;   // Sending gateway node ID (e.g. "!c1fc9198")
}
SettingMQTT Payload Content
encryption_enabled: false (default)Decoded MeshPackets. TEXT_MESSAGE_APP payloads are readable without channel key.
encryption_enabled: trueEncrypted MeshPackets. Requires the channel AES key to decrypt.

8. Node ID Format

Node IDs use the format !XXXXXXXX where XXXXXXXX is an 8-character lowercase hex string representing a 32-bit unsigned integer.

Node ID string:  !c1fc9198
Node number hex: 0xc1fc9198
Node number dec: 3254555032

# Parse in Python:
nodeNum = int(nodeIdString[-8:], 16)

# Format in Python:
nodeId = "!%08x" % nodeNum
Node IDNode NumberMeaning
!ffffffff4294967295Broadcast (all nodes)
!000000000Local/source node

The node number is derived from the last 4 bytes of the device MAC address.

9. Board Support

nRF52 RAK4631 No No No* No No — proxy only
Portduino Linux device simulator Sim Sim Yes Yes Yes

nRF52 Limitations

  • No WiFi or Ethernet — cannot connect to a broker directly.
  • Must use client proxy. The firmware rejects any configuration where enabled: true and proxy_to_client_enabled: false.
  • JSON not supported unless compiled with NRF52_USE_JSON flag.
  • The phone app handles all MQTT traffic for nRF52 radios.

ESP32 Notes

  • XIAO S3: Bluetooth is automatically disabled when WiFi is active (shared radio).
  • TLS uses setInsecure() — no certificate validation in firmware.
  • MQTT buffer: 1024 bytes (pubSub.setBufferSize(1024, 1024)).
  • Reconnect: 5 attempts, then triggers a WiFi reconnect cycle.
  • TCP connections are flagged as “EXPENSIVE” in firmware — retry interval is 30 seconds on failure.

10. Client Proxy Mode

When enabled (proxy_to_client_enabled: true)

  • The radio does not connect to the broker.
  • The phone app (Android or iOS) connects to the broker using the radio's credentials.
  • Broker client ID: MeshtasticAppleMqttProxy-!nodeId or MeshtasticAndroidMqttProxy-!nodeId.
  • The radio publishes through a proxy message queue to the phone app, which forwards to MQTT.

When disabled (proxy_to_client_enabled: false)

  • The radio connects directly to the broker using WiFi or Ethernet.
  • Broker client ID: !nodeId (for example, !c1fc9198).
  • The subscribe and publish operations both run inside the firmware.
  • The run loop polls at 200ms when connected, retries at 30 seconds on failure.
Common fault: If the radio shows correct MQTT config but never connects to the broker, check proxy_to_client_enabled first. When this is true, the radio's own MQTT client stays dormant and the phone app must proxy. No connection attempt appears in broker logs.

11. TLS

  • tls_enabled: true → port 8883.
  • tls_enabled: false → port 1883.
  • The firmware uses setInsecure() — no certificate validation.
  • TLS is not supported on nRF52. The firmware rejects tls_enabled: true on platforms without MQTT_SUPPORTS_TLS.

12. Mosquitto Broker Bridging

A Mosquitto bridge connects a local broker to a community mesh broker. The bridge forwards selected channel traffic in both directions.

Bridge Configuration Example

connection baymesh-bridge
address mqtt.bayme.sh:1883
topic msh/US/bayarea/2/c/MediumFast/# both 0
topic msh/US/bayarea/2/c/test/# both 0
topic msh/US/bayarea/2/json/# both 0
clientid gwyntel-bridge
remote_username meshdev
remote_password large4cats
start_type automatic
notifications true
notifications_local_only true
cleansession true
keepalive_interval 60

Per-Channel Bridge Exclusion

Mosquitto topic directives support MQTT wildcards (+, #) but do not support exclusions. To keep a channel private:

  1. Replace the catch-all # wildcard with explicit per-channel topic lines.
  2. Omit the private channel from the topic list.
  3. Channels that are not listed stay local to the broker.
# Forward LongFast, test, and JSON — but NOT mqtt or gwn channels:
topic msh/US/bayarea/2/c/LongFast/# both 0
topic msh/US/bayarea/2/c/test/# both 0
topic msh/US/bayarea/2/json/# both 0
Safety note: New channels are private by default. You must add a topic line to forward them. This is a feature, not a bug.

13. Community Broker Defaults

Usernamemeshdev

Default rootmsh

Community brokers use shared credentials. These are not IoT default passwords subject to CA SB-327. They are community credentials intentionally published for open mesh participation. Your own broker should use a randomly generated password.

Root topic is case-sensitive. msh/US/bayarea is not the same as msh/us/bayarea. If the node's root does not match the community broker's topic hierarchy, traffic silently goes nowhere.

14. Message Deduplication

MQTT QoS 1 guarantees at-least-once delivery. A Mosquitto bridge can echo messages, causing the same packet to arrive multiple times. To prevent duplicate processing:

  • Use the id field from the JSON uplink as a per-packet deduplication key.
  • Maintain a cache of recently seen packet IDs. A cache of 200 entries is sufficient.
  • Drop any packet whose id is already in the cache.

15. Map Reports

When map_reporting_enabled is true, the node publishes periodic map reports to {root}/2/map/.

Each report includes:

  • Node long name, short name, and ID
  • Position (with configurable precision) and altitude
  • Hardware model and role
  • Firmware version
  • LoRa region, modem preset, and primary channel name
  • Number of locally online nodes (last 2 hours, excluding MQTT-heard nodes)
SettingDefaultDescription
Position precision~1459m deviationObscures exact location.
Publish interval3600 seconds (1 hour)Minimum value. Cannot be set lower.

Map reports are unencrypted and visible on public maps. Available from firmware version 2.3.2.

17. Payload Size Limits

  • Data.payload.bytes — limited to approximately 240 bytes (firmware constant).
  • JSON downlink text — checked: jsonPayloadStr.length() <= sizeof(p->decoded.payload.bytes).
  • MQTT buffer — 1024 bytes (pubSub.setBufferSize(1024, 1024)).

18. Operational Pitfalls

proxyToClientEnabled after reboot. This setting can revert to true after a reboot or a batch config write. Always verify after reboot: 
meshtastic --host <ip> --info 2>&1 | grep proxyToClient
CLI socket error after config write. OSError: [Errno 9] Bad file descriptor is a known CLI bug. The config writes successfully before the error. Verify with --info.
PSK via CLI fails. --ch-set psk "TQ==" fails with expected bytes, str found. Use the Python API for custom PSK values.
--ch-set-url resets WiFi. Importing channels via URL resets network.wifi_ssid and network.wifi_psk. Re-apply WiFi settings after channel import. Also verify all channels imported — not all channels may transfer.
hopLimit resets to 7. lora.hop_limit can revert to the firmware default (7) after config writes. Always re-verify.
Do not modify radio MQTT config without explicit permission. Never change mqtt.address, mqtt.tls_enabled, mqtt.port, or mqtt.root on a working radio. If the radio fails to connect, check proxyToClientEnabled first.
Do not subscribe to unencrypted or public MQTT topics. Subscribe only to the specific channel topic (for example, msh/US/bayarea/2/json/mqtt/+). Never use the # wildcard to subscribe to all topics.
